## Getting Started

<Steps>
  <Step title="Get API Credentials">
    Register at the [Open Finance Sandbox](https://sandbox.openfinance.ae) to obtain:
    - Client ID
    - Client Secret
    - API Key
  </Step>

  <Step title="Authenticate">
    Use OAuth 2.0 to obtain access tokens:
    ```javascript
    const response = await fetch('/oauth/token', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        client_id: 'YOUR_CLIENT_ID',
        client_secret: 'YOUR_CLIENT_SECRET',
        grant_type: 'client_credentials'
      })
    });
    ```
  </Step>

  <Step title="Make API Calls">
    Include the Bearer token in all requests:
    ```javascript
    const data = await fetch('/api/accounts', {
      headers: {
        'Authorization': `Bearer ${token}`,
        'Accept': 'application/json'
      }
    });
    ```
  </Step>
</Steps>

## Bank Data Sharing APIs

Access customer account information, transaction history, and banking product data with proper consent.

### Key Endpoints

<AccordionGroup>
  <Accordion title="Account Information API" icon="wallet">
    **Base URL**: `/accounts`

    - `GET /accounts` - List customer accounts
    - `GET /accounts/{accountId}` - Get account details
    - `GET /accounts/{accountId}/balances` - Account balances
    - `GET /accounts/{accountId}/transactions` - Transaction history

    **Use Cases**: Personal finance apps, spending analytics, account aggregation
  </Accordion>

  <Accordion title="ATM Locator API" icon="map-pin">
    **Base URL**: `/atms`

    - `GET /atms` - Find nearby ATMs
    - `GET /atms/{atmId}` - ATM details and services
    - `GET /atms/search` - Search ATMs by location

    **Use Cases**: ATM finder apps, branch locators, service availability
  </Accordion>

  <Accordion title="Product Information API" icon="credit-card">
    **Base URL**: `/products`

    - `GET /products` - Available banking products
    - `GET /products/{productId}` - Product details
    - `GET /products/eligibility` - Check product eligibility

    **Use Cases**: Product comparison, eligibility checking, recommendations
  </Accordion>
</AccordionGroup>

## Bank Service Initiation APIs

Initiate banking services including payments, account opening, and foreign exchange.

### Key Endpoints

<AccordionGroup>
  <Accordion title="Payment Initiation" icon="money-bill-transfer">
    **Base URL**: `/payments`

    - `POST /payments` - Initiate payment
    - `GET /payments/{paymentId}` - Payment status
    - `PUT /payments/{paymentId}/authorise` - Authorize payment
    - `DELETE /payments/{paymentId}` - Cancel payment

    **Use Cases**: P2P payments, bill payments, merchant payments
  </Accordion>

  <Accordion title="Account Opening" icon="user-plus">
    **Base URL**: `/account-opening`

    - `POST /account-opening` - Start account application
    - `GET /account-opening/{applicationId}` - Application status
    - `PUT /account-opening/{applicationId}` - Update application

    **Use Cases**: Digital onboarding, account applications, KYC processes
  </Accordion>

  <Accordion title="Confirmation of Payee" icon="shield-check">
    **Base URL**: `/confirmation-payee`

    - `POST /confirmation-payee` - Verify payee details
    - `GET /confirmation-payee/{requestId}` - Verification result

    **Use Cases**: Payment verification, fraud prevention, account validation
  </Accordion>

  <Accordion title="FX Services" icon="coins">
    **Base URL**: `/fx`

    - `GET /fx/rates` - Current exchange rates
    - `POST /fx/quote` - Get FX quote
    - `POST /fx/transaction` - Execute FX transaction

    **Use Cases**: Currency exchange, international transfers, rate monitoring
  </Accordion>
</AccordionGroup>

## Authentication & Security

### OAuth 2.0 Flow

All APIs use OAuth 2.0 Client Credentials flow:

1. **Token Request**: Exchange credentials for access token
2. **API Access**: Include Bearer token in Authorization header
3. **Token Refresh**: Obtain new token when expired

### Security Best Practices

<Warning>
**Security Guidelines**:
- Never expose credentials in client-side code
- Use HTTPS for all API calls
- Implement proper error handling
- Validate and sanitize all inputs
- Store tokens securely
</Warning>

### Rate Limits

| Tier | Requests/Minute | Burst Limit |
|------|----------------|-------------|
| Sandbox | 100 | 150 |
| Production | 1000 | 1500 |

## Error Handling

Standard HTTP status codes and error responses:

```json
{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "The request is invalid",
    "details": "Missing required parameter: account_id"
  }
}
```

Common error codes:
- `400` - Bad Request
- `401` - Unauthorized (invalid token)
- `403` - Forbidden (insufficient permissions)
- `429` - Too Many Requests (rate limited)
- `500` - Internal Server Error
